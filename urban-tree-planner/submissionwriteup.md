# Submission Write‑up

## Problem Statement

Urban planners and city officials need an intelligent assistant that can quickly answer domain‑specific queries such as "What’s the current weather in San Francisco?" or "What time is it in Tokyo?" while ensuring that any request complies with security policies (e.g., no disallowed external calls). Existing tools require manual scripting or multiple APIs, which is error‑prone and hard to maintain. This project builds a lightweight, privacy‑preserving agent that routes user requests to the appropriate sub‑agents (weather, time) and enforces a security checkpoint before any external call.

## Solution Architecture

![Solution Architecture](./docs/solution_architecture.png)

The architecture consists of:
- **Orchestrator LlmAgent** – receives user prompts, sanitises input via `security_checkpoint`, and delegates to the appropriate tool.
- **WeatherAgent & TimeAgent** – simple LlmAgents that wrap external API calls.
- **Security Checkpoint** – a custom function wrapped as an `AgentTool` that validates request content.
- **MCP Server** – hosts the agents‑cli tooling for local development and testing.
- **ADK Playground** – UI for interactive testing of the orchestrator.

All components live under the `urban-tree-planner` workspace, making the solution easy to run with a single `make playground` command.

## Security Design
| Control | Why it matters |
|---|---|
| Input sanitisation (`security_checkpoint`) | Prevents malicious payloads or disallowed URLs before any external request. |
| Role‑based tool exposure | Only the orchestrator can call the sub‑agents; other agents cannot invoke arbitrary tools. |
| In‑memory credential service (development) | Guarantees no secret leakage in the local environment. |
| Logging of all tool calls | Provides an audit trail for debugging and compliance. |

## MCP Server Design
| Tool | Purpose |
|---|---|
| `weather_agent` | Returns current weather via OpenWeatherMap API (mocked for demo). |
| `time_agent` | Returns current time for a given timezone. |
| `security_checkpoint` | Validates request JSON for allowed actions. |
| `orchestrator` | Routes user queries to the appropriate tool after validation. |

The MCP server registers each tool using `AgentTool` so that the playground UI can discover and invoke them automatically.

## Human‑in‑the‑Loop (HITL) Flow
1. **User** submits a natural‑language query in the ADK Playground UI.
2. **Orchestrator** receives the request and invokes `security_checkpoint`.
3. If the request passes, the orchestrator selects the appropriate sub‑agent.
4. The sub‑agent performs the request and returns a structured response.
5. **User** sees the answer in the UI; if the security checkpoint rejects the request, a friendly error is shown.

## Demo Walkthrough (Sample Test Cases)

2. **Time query** – Input: `{ "intent": "time", "zone": "Asia/Tokyo" }`. Expected: Orchestrator calls `time_agent` and returns the current time in Tokyo.

## Impact / Value Statement
The agent provides a **quick, secure, and extensible** interface for common city‑planning queries. It reduces development effort for municipal apps, improves data reliability by centralising API calls, and demonstrates a pattern that can be expanded to other civic services (traffic, zoning, public services). By exposing a clear security layer, the solution is suitable for production environments that must comply with strict data‑handling policies.
